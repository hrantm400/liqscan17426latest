# План: Bias «0» и SuperEngulfing «только N live»

## Как устроено сейчас (коротко)

### Monitor Bias (`MonitorBias.tsx`)

- Данные: `useMarketData({ strategyType: 'ICT_BIAS', limit: 5000 })` + опционально подмешивание **`live-bias`** (пересчёт bias по свечам).
- Карточки **4H / 1D / 1W** считают сигналы из **`statusFilteredSignals`** при вкладке **LIVE**.
- Вкладка **LIVE** (`useLifecycleFilter`): для **ICT_BIAS** используется **`lifecycleStatus === 'PENDING' || 'ACTIVE'`** (не поле `state`, как у SuperEngulfing).
- Отдельно: счётчики в табах (`statusCounts`) режут выборку по **≥ $20M** 24h объёма (`globalValidSignals`), когда загружен `volumeMap` с Binance.

### Monitor SuperEngulfing (`MonitorSuperEngulfing.tsx`)

- Запрос: `useMarketData({ strategyType: 'SUPER_ENGULFING', limit: 1000, minVolume: 20_000_000 })` — на бэкенде сначала выборка из БД, потом фильтр по объёму.
- Вкладка **LIVE**: для SE v2 — **`signal.state === 'live'`**; иначе legacy `PENDING` / `ACTIVE`.

---

## Почему может быть Bias = 0 на карточках

| Причина | Как проверить |
|--------|----------------|
| На вкладке **LIVE** нет строк с `PENDING`/`ACTIVE` (все закрыты/архив) | В БД: распределение `lifecycleStatus` для `ICT_BIAS`. |
| Сигналы есть только в таймфреймах **вне** 4h/1d/1w | Проверить `timeframe` у `ICT_BIAS` в БД. |
| **`live-bias`** отдаёт пусто/502 — метки на карточках считаются не из live-bias, но список может выглядеть «пустым» при ошибках API | Network: `/api/signals/live-bias?...`. |
| Жёсткий фильтр **$20M** в счётчиках табов при узком наборе пар | Сравнить число на вкладке **ALL** vs **LIVE**. |

---

## Почему SuperEngulfing «всего 5 live»

| Причина | Как проверить |
|--------|----------------|
| В БД реально **5** строк SE с **`state = 'live'`** после сканера | SQL / админка по `SUPER_ENGULFING` + `state`. |
| Порог **20M** отсекает почти все пары | Временно сравнить запрос **без** `minVolume` (или меньший порог) — только для диагностики. |
| Сканер реже создаёт новые «live» или быстро переводит в `closed` | Логи `ScannerService`, расписание сканов. |

## План работ (по приоритету)

### Фаза 1 — Диагностика (1–2 часа)

1. **SQL / Prisma**:  
   - `ICT_BIAS`: count по `timeframe`, по `lifecycleStatus`.  
   - `SUPER_ENGULFING`: count по `state`, по `timeframe`.
2. **Сравнить UI**: вкладка **ALL** на Bias — есть ли строки при **0** на карточках LIVE. Если в ALL есть, а в LIVE нет — проблема в **lifecycle**, не в «отсутствии данных».
3. Проверить **production**: нет ли стабильных **502** на `/api/signals` и `/api/signals/live-bias` (иначе фронт показывает пусто).

### Фаза 2 — Продукт / UX (по согласованию)

1. **Bias**: если бизнес-логика «одна актуальная запись на symbol+tf» и она не всегда `PENDING`/`ACTIVE`, то либо:
   - выставлять при upsert статусы, согласованные с вкладкой LIVE, **либо**
   - на странице Bias для вкладки LIVE использовать критерий ближе к продукту (например последняя запись по времени независимо от статуса — **только если так задумано**).
2. **Подсказка в UI**: при пустом LIVE — баннер «Нет активных сигналов, см. ALL / CLOSED» (уменьшит ощущение «сломалось»).
3. **Объём 20M**: вынести в настройку tier / переключатель «Показать все ликвидности» для внутренней проверки (не обязательно для публики).

### Фаза 3 — Бэкенд / сканер (если данные в БД «не те»)

1. Проверить **`checkICTBias`** / upsert: какие **`lifecycleStatus`** пишутся при обновлении одной и той же пары.
2. Проверить **SuperEngulfing v2**: когда `state` переходит `live` → `closed`, не слишком ли агрессивно.
3. При необходимости — смягчить или пересмотреть **`minVolume`** на уровне API для списков мониторов (отдельно от дашборда).

### Фаза 4 — Наблюдаемость

1. Метрика: число сигналов по стратегии и вкладке (лог раз в N минут или простой health-эндпоинт для админов).
2. Алерт при **0** ICT_BIAS live дольше X часов (опционально).

---

## Быстрые проверки без кода

- Открыть **Bias** → вкладка **ALL**: есть ли строки?  
- Открыть **SuperEngulfing** → **ALL** vs **LIVE**: совпадает ли с ожиданиями по числу строк.

Если **ALL** полон, а **LIVE** почти пуст — это ожидаемо при текущих правилах lifecycle/state, а не «пропажа данных».
